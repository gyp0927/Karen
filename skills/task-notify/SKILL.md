# Task Notify — 任务完成桌面通知

在完成任务后，通过 Windows 桌面通知告知用户结果。通知内容根据实际工作内容动态生成摘要。

## 触发时机

以下情况应调用本 skill：

- **用户主动要求**："通知我一下"、"弹窗提醒"、"完成后提醒我"
- **代码工作流完成**：修改代码 → 运行测试 → 提交 → 推送，整个流程结束后
- **长时间任务完成**：运行了耗时的测试、构建、数据处理等操作后
- **多步骤任务结束**：完成了一系列文件修改、配置调整等操作后
- **需要用户关注的结果**：测试失败、推送冲突、需要用户决策的情况

## 使用方法

### 发送通知

调用通知脚本，传入动态生成的摘要信息：

```bash
python skills/task-notify/scripts/notify.py -t "Claude" -m "通知内容摘要" -d 5
```

参数说明：
- `-t, --title`：通知标题（默认 "Claude"）
- `-m, --message`：通知正文，应简洁概括完成的工作（默认 "任务已完成"）
- `-d, --duration`：显示时长秒数（默认 5 秒）

### 动态摘要生成规则

通知消息应简洁（30 字以内最佳，最多不超过 80 字），概括核心结果：

**代码修改类任务**：
- 好：`"代码修复完成：已修复 web/app.py 导入错误，测试全部通过"`
- 好：`"功能添加完成：新增用户登录接口，3 个测试通过"`
- 差：`"我已经完成了你要求的工作，现在可以开始下一个任务了"`（太泛）

**测试/构建类任务**：
- 好：`"测试完成：12 通过 / 0 失败，已推送至 main"`
- 好：`"构建失败：第 47 行语法错误，需修复"`

**文件操作类任务**：
- 好：`"文件创建完成：已生成 report_2024.xlsx"`
- 好：`"配置更新完成：API Key 已轮换，服务重启正常"`

**多步骤复杂任务**：
- 只报告最终状态和关键结果，省略中间步骤
- 如有失败项，简要说明原因

### 通知时机判断

不要每次小操作后都弹通知，只在以下情况触发：

| 场景 | 是否通知 |
|------|----------|
| 单次文件读取/查看 | 否 |
| 修改 1-2 个文件（未要求后续操作） | 可选 |
| 修改代码 + 运行测试 + 提交推送（完整工作流） | **是** |
| 长时间运行任务（>30 秒）结束 | **是** |
| 测试失败/构建失败需要用户处理 | **是** |
| 用户明确要求通知 | **是** |
| 多轮对话探索阶段 | 否 |
| 最终交付/总结阶段 | **是** |

## 完整示例

用户要求："修复 web/app.py 的导入错误，然后测试并推送"

Claude 执行步骤：
1. 修复导入错误（编辑文件）
2. 运行测试 `python test_all.py`
3. 测试通过
4. 提交 `git add -A && git commit -m "Fix: ..."`
5. 推送 `git push ...`
6. **发送通知**：

```bash
python skills/task-notify/scripts/notify.py -t "Claude" -m "代码修复完成：web/app.py 导入错误已修复，测试全部通过，已推送至 main" -d 5
```

## 故障排除

| 问题 | 解决方法 |
|------|----------|
| 通知不显示 | 检查 Windows 通知设置中是否禁用了应用通知 |
| 脚本运行报错 | 确保是 Windows 系统且 PowerShell 可用 |
| 通知内容过长被截断 | 控制消息在 80 字以内 |
| 通知一闪而过 | 增加 `-d` 参数，如 `-d 10` |
